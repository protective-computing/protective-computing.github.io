Protective Computing Audit Checklist

A structured checklist for auditing software against Protective Computing requirements. Use this before launch, during review, or when evaluating a third-party system.

This checklist maps to the six Protective Computing principles. Each section identifies failure modes specific to that principle and asks whether your system addresses them. The goal is not compliance theatre — it is honest pressure-testing against degraded and adversarial conditions.

For the normative specification these checks derive from, see the Protective Computing Specification v1.0.

Before You Start

Identify your threat context:

  • Who are your most vulnerable users? (Users in crisis, under surveillance, with low-resource devices?)
  • What is the worst-case operating environment? (No network, low power, compromised device, coercion?)
  • What is the critical path? (The minimum the user must be able to accomplish to remain safe?)
  • Who could be adversarial? (Abusers, institutions, attackers, or your own infrastructure?)

Answer these first. The checklist below has more weight when mapped to your specific threat context.

1. Reversibility

Principle 1

Can users undo or recover from actions? Full reference →

  • Destructive actions (delete, send, submit) can be undone within a reasonable window
  • Users are warned before irreversible operations with a clear description of what cannot be recovered
  • Deletion is soft-deleted first; hard deletion is delayed or requires explicit confirmation
  • Export is available so users can retrieve their data before leaving
  • Session loss (crash, power cut, connectivity drop) does not silently discard user work
  • Cascading destructive operations are blocked or require explicit acknowledgement of downstream effects

2. Exposure Minimization

Principle 2

Is the data surface as small as possible? Full reference →

  • Data collection is justified field by field — no "collect now, decide later" fields exist
  • Sensitive data is encrypted at rest with user-controlled or per-user keys
  • Data is not retained beyond its operational purpose; retention schedules are enforced, not documented
  • Logs do not capture sensitive payloads (form values, message bodies, health data)
  • Third-party integrations are audited for what they receive; unnecessary sharing is removed
  • Transmission uses TLS; certificates are validated; no plaintext fallbacks exist
  • Aggregate or anonymous data is used in place of identified records wherever possible

3. Local Authority

Principle 3

Does the user retain control offline and without your infrastructure? Full reference →

  • Critical user functions work without a network connection
  • Data is cached locally so users are not locked out when your servers are unavailable
  • Authentication does not block access to locally cached content
  • Sync conflicts are surfaced to the user rather than silently resolved by the server
  • Users can export all their data in a portable format without contacting support
  • The system does not require a continuous server heartbeat to remain usable

4. Coercion Resistance

Principle 4

Can users protect themselves under duress or forced disclosure? Full reference →

  • Users can delete or wipe sensitive data rapidly, including under duress
  • The system does not store information that could be used to harm the user if extracted
  • Strong authentication options are available (passphrase, hardware key) not just biometrics or SMS
  • There is no hidden recovery backdoor that law enforcement or abusers could exploit
  • The system does not reveal that data has been wiped (plausible deniability where applicable)
  • Security claims in documentation are accurate — no false promises of protection

5. Degraded-Infrastructure Functionality

Principle 5

Does the system work under low resources? Full reference →

  • The critical path has been tested on 2G or simulated low-bandwidth connections
  • The system does not assume full battery; power-intensive features are optional, not default
  • Page or screen weights are measured; heavy assets are lazy-loaded or omitted on constrained connections
  • The UI degrades gracefully: core functionality survives without JavaScript, WebSockets, or CDN assets
  • Keyboard-only navigation is supported on all critical flows
  • Low-RAM devices (512 MB or less) can complete core tasks without crashes
  • Animations and background processes are optional or suppress-able to conserve resources

6. Essential Utility

Principle 6

Is the system optimized for survival tasks, not engagement? Full reference →

  • The critical path is documented and is the fastest, most reliable flow in the system
  • Dark patterns (forced continuity, deceptive defaults, guilt-based friction) are absent
  • Notifications and alerts are proportionate to urgency — no engagement-bait prompts
  • Features that compete for user attention without providing utility have been removed or made opt-in
  • Cognitive load on critical flows is minimized: fewer decisions, shorter text, progressive disclosure
  • The system does not require or exploit emotional or cognitive vulnerability to achieve its goals

After the Audit

For each item that fails:

  • Classify severity: Does this failure block the critical path, or is it a marginal risk?
  • Assign ownership: Who is responsible for fixing it?
  • Set a remediation target: Block launch, fix in next sprint, or track as known debt?
  • Document it: Undocumented known risks become silent liabilities.
Formal Compliance Levels

The Protective Computing Specification v1.0 defines four compliance levels (Level 1–4), with MUST/SHOULD/MUST NOT requirements and formal verification expectations. Use that spec for compliance claims. Use this checklist for practical design review.
